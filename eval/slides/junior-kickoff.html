The Project

What we're building

An AI family history consultant — a research assistant that helps people do real genealogy.

It runs inside Claude Cowork, a desktop app for Claude.

We make it good at genealogy by adding two things…

Skills & Tools

~2 dozen Tools

Connect to outside data — search records, read a person's tree, look up places, query the FamilySearch wiki.

~2 dozen Skills

Do the genealogy work — research planning, record extraction, locality guides, conflict resolution, and more.

The skills are where the genealogical expertise lives. Making them better is the whole game.

How we make the AI better

To improve a skill, we first have to measure how well it works in many different situations.

So we built a set of “tests.” Each test captures one realistic moment and asks: did the AI do this well?

A test is graded by another AI (the “judge”) — fast, but not reliable on genealogical judgment. That's where you come in: your corrections fix the grades and teach the judge to grade better over time.

What's inside one test

  • A research scenario — what's been found so far in this project
  • The user's input — what the person asks the AI to do
  • Tool responses — what the tools returned
  • The AI's response — what the skill actually produced
  • Judge context — instructions for how to grade the response
  • Response grades — the judge AI's scores, which you review

The grading vocabulary

Every test is graded on several dimensions:

  • Base dimensions (always): Correctness, Completeness, Tool Arguments
  • Rubric dimensions: 3–5 skill-specific checks

Each dimension gets one score:

1 fail    2 partial    3 pass

(Tool Arguments can also be N/A when the skill made no tool calls.)

Grading well

  • Read the full trace before scoring — input, tool calls, output.
  • Judge against the rubric, not your overall gut feeling about the skill.
  • Use 2 (partial) honestly — it isn't all-or-nothing.
  • When you disagree with the AI, write one line saying why.
  • No trick questions — grade the way your expertise tells you.

Reviewing Scores
in the App

The review app

  • A small web app that runs on your own machine.
  • Launch it with Start.bat — a browser tab opens automatically.
  • Go to localhost:3000/results — run logs grouped by skill.
  • Click your skill's latest run to open it.

Keep the black command window open the whole time — closing it stops the app.

Each test shows two blocks

① The Trace

  • The skill's input
  • The scenario (what's been found so far)
  • Tool calls + what the tools returned
  • Claude's final output

② The Grades

  • One row per dimension
  • The judge's score 123
  • The judge's rationale
  • A picker for your correction

Three things you'll do per dimension

SituationWhat to do
You agree with the judgePick the same score; leave the comment empty.
You disagreePick a different score + write a one-line reason.
The whole test is rightClick “Agree with all” on the test header — marks every dimension reviewed in one click.

Agreeing still counts as reviewing — you must touch every dimension.

Speeding up & finishing

Keyboard (when a picker is focused)

  • 1 2 3 — set the score
  • Tab / Shift+Tab — move between dimensions
  • ? — show shortcut help

The completeness counter

  • Header shows e.g. 8 / 12 reviewed
  • You're done when it reads N / N
  • Your PR won't pass until every dimension is reviewed

Corrections save automatically — you'll see “saved” in the header.

A fast, honest review pattern

  • Scan the trace first — understand what the skill was asked and what it did.
  • For each dimension, check the judge's rationale against the rubric.
  • “Agree with all” on tests that look clearly right…
  • …and spend your attention on the ones you actually disagree with.
Quality over speed — but you don't have to re-justify every correct grade.
