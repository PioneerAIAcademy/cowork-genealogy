Creating, Running &
Evaluating E2E Tests

E2E means "end-to-end" — a test that asks the AI to do a whole piece of real research, start to finish. A 30-minute orientation; the step-by-step how-to is the "E2e tests" section of eval/README.md.

What an e2e test is

  • Snapshot a real, well-researched FamilySearch tree.
  • Hide the answer — strip a focused fact (e.g. who were the parents?).
  • The AI must recover it from records, via /research.
  • A judge grades the result: pass / partial / fail.
The tree-reading tools are blocked during the run, so the AI can't just look the answer up on FamilySearch. A pass means it researched it — not that it peeked.

The work is a loop

Create→ Run→ Interpret→ Grade→ Improve skill→ Re-run↺ PR

Run it, see how the skill did, improve the skill, and re-run — until you can't improve the result anymore. Then submit a PR.

One run ≈ 20–60 minutes and $3–10. Run one at a time, and plan your day around a handful — not dozens.

Setup

Once per machine

Windows
Setup.bat
macOS / Linux
make install
echo 'ANTHROPIC_API_KEY=sk-...' >> eval/.env

Each day, before running

Windows
Login.bat        & CheckSetup.bat
macOS / Linux
make e2e-login   && make e2e-preflight

Log in once a day (~24h FamilySearch token). The ready-check ("preflight") confirms login, server, key & tools before you spend time and money on a run.

Create — /author-e2e-fixture

Run it in Claude Code at the repo root. Two ways to give it the data — the subject must be deceased either way, and it does the writing for you:

From a FamilySearch PID

Give it a well-researched person ID. It reads the tree, you pick what to hide, and writes the files.

From a research document

No FamilySearch access? Hand it a report, research log, or proof article — it builds the starting tree by hand from the document and uses the document's conclusion as the answer.

Either way, keep it focused (one question, 1–5 findings), then confirm the answer is really gone: ValidateFixture.bat / make e2e-validate TEST=<slug>.

Run a test

Windows
RunE2E.bat
# prompts for the slug
macOS / Linux
make e2e-run TEST=kenneth-quass-death

Writes the result, a readable transcript, and the AI's final tree to eval/runlogs/e2e/<slug>/.

Rebuilds the server first if needed. Remember — one run at a time.

See what the AI did

Start with /interpret-e2e-result for the headline verdict. Then read the step-by-step — every search, result, and skill it used is in run-<ts>.transcript.md.

Ask Claude about the run

"Did it skip a search it should have run? Try maiden-name or spelling variations? Call a finding direct when the evidence is only indirect? Where and why did it stop?"

This is where you catch the missed search or the over-claim — the verdict alone won't show it.

View the result in the app

Windows
ViewE2E.bat
macOS / Linux
make e2e-view TEST=<slug>

Loads the latest run into the Research Viewer — the final tree, the research log, conflicts, and each finding's direct vs indirect badge.

Open eval/.e2e-view in the viewer once (its Open Project button) and keep it open — every re-run, just run e2e-view again and it refreshes live.

Grade the result

  • For each expected finding, decide whether the AI really found it: true / partial / false.
  • Ask Claude Code to grade the run — it records your labels in run-<ts>.ann.json, saved beside the run log.
Grading is squarely genealogy judgment — and that .ann.json is the grade you commit. Where you disagree with the AI judge, your label is what counts.

Improve the skill → re-run → compare

A "skill" is a page of plain-English instructions telling the AI how to research (a SKILL.md file).

  • Diagnose — what did the AI get wrong, and what should good research have done?
  • Adjust the skill's instructions, then re-run (picked up automatically).
  • Did it improve? Repeat until you can't improve it anymore.
Re-run before concluding — the AI isn't perfectly repeatable; one finding flipping may be chance.

Submit a PR

Commit together:

  • the fixture + a passing run log
  • your grade (run-<ts>.ann.json) + any skill improvements
The maintainers review and merge. You create and push; you don't merge your own.
