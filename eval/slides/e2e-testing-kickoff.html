Creating, Running &
Evaluating E2E Tests

E2E means "end-to-end" — a test that asks the AI to do a whole piece of real research, start to finish. A 30-minute orientation; the step-by-step how-to is the "E2e tests" section of eval/README.md.

What an e2e test is

  • Snapshot a real, well-researched FamilySearch tree.
  • Hide the answer — strip a focused fact (e.g. who were the parents?).
  • The AI must recover it from records, via /research.
  • A judge grades the result: pass / partial / fail.
The tree-reading tools are blocked during the run, so the AI can't just look the answer up on FamilySearch. A pass means it researched it — not that it peeked.

Two speeds: a fast loop, one slow score

Create→ Debug live in Cowork⟳ tweak the skill→ Headless run + grade→ PR

Iterate fast, live: watch /research in Cowork, tweak the skill, run again — minutes per loop, no waiting for a verdict.

Score once, at the end. When it works live, do one headless run — the honest, tree-read-blocked check — then grade that run. A 20–60 min / $3–10 run belongs at the end, not after every tweak.

Setup

Once per machine

Windows
Setup.bat
macOS / Linux
make install
echo 'ANTHROPIC_API_KEY=sk-...' >> eval/.env

Each day, before running

Windows
Login.bat        & CheckSetup.bat
macOS / Linux
make e2e-login   && make e2e-preflight

Log in once a day (~24h FamilySearch token). The ready-check ("preflight") confirms login, server, key & tools before you spend time and money on a run.

Create — /author-e2e-fixture

Run it in Claude Code at the repo root. Two ways to give it the data — the subject must be deceased either way, and it does the writing for you:

From a FamilySearch PID

Give it a well-researched person ID. It reads the tree, you pick what to hide, and writes the files.

From a research document

Alternatively, hand it a report, research log, or proof article — it builds the starting tree by hand from the document and uses the document's conclusion as the answer.

Either way, keep it focused (one question, 1–5 findings), then confirm the answer is really gone: ValidateFixture.bat / make e2e-validate TEST=<slug>.

Debug live in Cowork — start here

Windows
SeedProject.bat
macOS / Linux
make e2e-project TEST=<slug>
  • Seeds an editable project from the fixture's starting state. Open it in Claude Cowork, run /research — init-project is auto-skipped.
  • Open the same folder in the Research Viewer: watch the log, assertions, and conflicts appear live, and ask Claude "why didn't you search X? why direct, not indirect?"
Watch HOW it gets each answer — by searching records, not by reading the tree. Live runs don't block tree-reads, so a "find" via the tree won't hold up when you score.

Tweak the skill, re-run live — repeat

A "skill" is a page of plain-English instructions telling the AI how to research (a SKILL.md file).

  • Diagnose what went wrong, reword the skill's instructions, and run /research again in Cowork.
  • Edits are picked up on the next run — watch the fix work in minutes, no waiting for a verdict.
  • Loop until it researches well and gets there by searching records.
This is the fast inner loop — stay here, iterating live, until you're confident. No headless runs yet.

Score it — once, at the end

Windows
RunE2E.bat
macOS / Linux
make e2e-run TEST=<slug>

When it works live, do one headless run — the honest check that blocks the tree-read tools. It writes the result + transcript + final tree to eval/runlogs/e2e/<slug>/; then /interpret-e2e-result for the verdict.

If it fails, the live fix didn't hold — read run-<ts>.transcript.md (or make e2e-view) to see what the blocked run did, then back to live debugging. One run is noisy — re-run a borderline result before trusting it.

Grade the run

  • For the run you'll commit, decide for each finding: true / partial / false.
  • Ask Claude Code to grade the run — it records your labels in run-<ts>.ann.json, beside the run log.
Grading is squarely genealogy judgment — and that .ann.json is the grade you commit. Do it once, on the passing run, not every iteration.

Submit a PR

Commit together:

  • the fixture + a passing run log
  • your grade (run-<ts>.ann.json) + any skill improvements
The maintainers review and merge. You create and push; you don't merge your own.
