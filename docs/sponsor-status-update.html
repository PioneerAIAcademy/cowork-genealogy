Project Overview

An AI family-history consultant that helps researchers — beginners through Certified Genealogists — work the Genealogical Proof Standard (GPS): plan, search, weigh evidence, resolve conflicts, and write defensible conclusions.

It assists; it does not replace the researcher. Every record, match, and suggestion is a starting point to verify against original sources. The GPS requires the researcher — not the tool — to reach conclusions. Outputs are working drafts, not citable conclusions.

What we ship

  • MCP server — runs on the host with full network access; wraps FamilySearch, Wikipedia, and our reference APIs as 31 tools
  • Cowork plugin — 26 skills that run in the sandboxed VM and teach Claude how to apply the tools
  • Plus a hosted web workbench and an eval harness for regression testing

Standards & audit trail

  • GPS — the 5-question framework every conclusion must pass
  • Evidence Explained — Mills citation standard
  • research.json — the GPS audit trail (how we concluded)
  • tree.gedcomx.json — the deliverable (what we concluded)

Architecture — One Engine, Many Front Doors

"Does this need the network?"  Yes → it's an MCP tool on the host (full internet). No → it's a skill in the sandbox (data, formatting, templating).

Host — full network access

  • MCP server — the 31 tools
  • Calls FamilySearch OAuth, the Wiki RAG sidecar, and the population-stats sidecar
  • The network lives here because the sandbox can't reach arbitrary domains

Sandbox — no network egress

  • Plugin skills + stdlib Python — the 26 skills
  • Read/write the two project files — research.json, tree.gedcomx.json
  • Reaches data only by calling tools — structured JSON in, JSON out

All three delivery modes run this same engine — in Claude Cowork's VM, or in a per-user cloud sandbox for the REST API and the Client-Server app. Build once; ship everywhere.

Three Delivery Modes

Same engine, three front doors — each aimed at a different audience. All three available by end of month.

Claude Cowork + Companion App Individuals

The desktop extension (.mcpb) and plugin run in Claude Cowork; a desktop companion app shows the live project viewer as the research unfolds. The path for individual researchers and the in-house team.

REST API for Help Team Integrators

A dedicated /v1 REST chat surface with bearer-key auth lets the Help team's chatbot drive the agent over plain REST — create a session, send a message, get the reply — with none of our browser machinery.

Client-Server App Alpha testers

A hosted, browser-based workbench — login, chat sidebar, and the same live project viewer — running a sandboxed per-user agent. The vehicle for alpha testing with no install.

One shared engine behind all three means a fix or a new skill lands everywhere at once — no per-channel reimplementation.

Unit Testing Strategy

How a genealogist + developer team takes one skill from a coverage gap to a merged, regression-proof improvement.

  1. Genealogist + developer teams identify an area that isn't covered by existing tests (eventually they will use e2e test results and alpha test feedback for this step)
  2. They create new unit tests, along with fixtures to mock tool calls, to cover this area
  3. They run the tests through the test harness and review the results
  4. An LLM-judge grades the test results, and they review and correct the LLM-judge's grades
  5. If the skill fails the tests, they improve the skill until it passes the tests
  6. They run the harness over all unit tests for the skill and review and correct all LLM-judge grades to ensure no regressions
  7. They submit the new unit tests, skill changes, test harness runlog, and grade corrections as a PR for senior genealogists to review
  8. Senior genealogists either review the PR or request changes
  9. Eventually we will use the LLM-judge grade corrections to calibrate the LLM-judge to be more-aligned with the human judges

Why this shape: the LLM-judge makes grading scale to hundreds of tests; the human grade corrections keep it honest — it's measured against expert grades, not trusted blindly; and calibrating the judge against those corrections makes it more like a senior genealogist over time. The harness, run-log versioning, and review UI are built and in use; test authoring is ramping now.

Quality: End-to-End Tests

Unit tests check one skill in isolation. End-to-end tests run the whole system — the full plugin + MCP server flow on a complete research project, from a FamilySearch person through to a written conclusion.

Same pipeline as unit tests

  • Submitted by junior genealogist + developer pairs
  • Reviewed by senior genealogists — starting beginning of July
  • Scored by an LLM judge, then corrected by humans
  • Calibrated over time as corrections accumulate

What an e2e case measures

  • Take a well-researched tree, strip out the answer, record what was removed as expected findings
  • Run the agent end-to-end and measure how much of the answer it recovers
  • Catches failures a single-skill test can't — wrong routing between skills, lost state, a weak hand-off

The authoring and interpretation tooling exists; senior review of e2e cases begins early July.

Alpha Testing — Real-World Use

Tests catch regressions; only real research catches what tests can't imagine. Alpha puts the product in expert hands and captures structured feedback.

Formal feedback, built in

  • A one-click feedback mechanism in both front ends — the Cowork companion app and the Client-Server app
  • Each submission bundles the project state, so we can reproduce exactly what the researcher saw
  • Feedback feeds straight into the test pipeline — a real-world miss becomes a regression test

Who, and when

  • Senior genealogists doing real research — starting with Robert Raymond
  • Debbie's staff join in a later wave
  • Kicks off mid-July

Expert eyes are the quality gate no automated test can replace — does this actually help a researcher do better genealogy?
