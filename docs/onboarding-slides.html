Project Goal

Build an AI family history consultant — an AI that helps you do genealogy research the right way, following the Genealogical Proof Standard.

Important distinction: This is not an AI that does your genealogy for you. Every record returned, every match suggested, every conflict resolution is a starting point for you to verify against original sources. The GPS requires the researcher — not the tool — to weigh evidence, resolve conflicts, and reach conclusions. Outputs are working drafts in your research process, not citable conclusions.

Who it's for

  • Beginners starting their first family tree
  • Experienced self-taught researchers
  • Certified Genealogists preparing peer-review-quality cases
  • The standard is the same for all — the tools help you meet it faster

The standard we follow

  • GPS — Genealogical Proof Standard: the 5-question framework every genealogical conclusion must pass
  • Evidence Explained — citation standard by Elizabeth Shown Mills
  • GPS audit trail built into every project file

Project Overview

Two tightly coupled artifacts ship from this single repo, plus a hosted web workbench (POC) and an eval harness.

MCP Server Runs on your Mac/PC

TypeScript server (packages/engine/mcp-server/) packaged as a Claude Desktop Extension (.mcpb). Has full network access — calls FamilySearch, Wikipedia, and our hosted sidecar APIs. Exposes 31 tools to Claude.

Cowork Plugin Runs in sandboxed VM

Skills, templates, and Python scripts (packages/engine/plugin/) packaged as a .zip. Runs inside Cowork's Linux VM with no direct internet access. Communicates with the MCP server exclusively via tool calls — JSON in, JSON out.

Web Workbench POC

Browser version: chat sidebar + live project viewer. React+Vite frontend (apps/web/), FastAPI control plane (apps/server/), shared viewer component (packages/viewer-ui/). Runs fully on mocks — no cloud credentials needed.

Eval Harness

Skill regression tests under eval/. Unit tests check individual skill logic; end-to-end tests run the full plugin + MCP server flow. A CRUD UI lets you browse, create, and manage test cases without touching files. You'll be writing test scenarios for the eval harness this week.

Architecture — The Key Rule

"Does this need the network?"
If yes → it's an MCP tool (runs on the host). If no → it can be a skill script (runs in the VM).

Host (your machine)

  • MCP Server — full internet access
  • Calls FamilySearch OAuth APIs
  • Calls our hosted Wiki RAG sidecar
  • Calls our hosted population-stats sidecar
  • Claude Desktop Extension format (.mcpb)

Skills call tools via MCP tool calls — the only communication channel between the two sides.

Cowork VM (sandboxed)

  • Plugin skills & Python scripts
  • Reads/writes project files:
    research.json — GPS audit trail
    tree.gedcomx.json — the deliverable
    and search-result sidecar files in results/
  • No network egress — can't call APIs directly
  • No shared files with the host at runtime

This is why Python skill scripts must use stdlib only — no pip installs, no HTTP calls.

Project Files

Every research project lives in two files. Skills read from and write to these files; there is no other shared state.

research.json — The GPS Audit Trail

Contains everything that records how you reached your conclusions — not just what you found, but why you believe it.

  • Research questions and their status
  • Research plans (what collections to search, why)
  • Search log entries (what was searched, what was found)
  • Evidence assertions extracted from records
  • GPS evidence classifications (Primary/Secondary, Direct/Indirect/Negative)
  • Conflict notes and resolutions
  • Hypotheses and competing candidates
  • Proof summaries and exhaustiveness declarations
  • Researcher profile (experience level, subscriptions)

tree.gedcomx.json — The Deliverable

The family tree itself — what you've concluded. Simplified GedcomX format (snake_case).

  • Resolved persons (names, dates, places, facts)
  • Relationships (couples, parent-child)
  • Source citations attached to facts
  • Only updated after a conclusion reaches "Probable" or higher confidence
  • Validated against a strict schema after every write

These two files are the entire persistent state of a project. Cowork sessions are ephemeral — only the project folder survives across sessions.

The GPS Research Workflow

The plugin follows the Genealogical Proof Standard — a systematic cycle, not a one-shot search. Skills map directly to each step.

1
Start the project init-project → seeds tree from FamilySearch person
2
Pick a research question question-selection → ranks open questions by value
3
Plan the research research-plan → sequence of record sets, rationale, fallbacks
4
Execute searches search-records · search-full-text · search-external-sites
5
Extract evidence record-extraction → atomic assertions from each record
6
Classify evidence record-extraction → Primary/Secondary, Direct/Indirect/Negative (first-and-final at extraction)
7
Polish citations citation → Evidence Explained standards
8
Link assertions to persons person-evidence → identity evaluation with threshold enforcement
9
Build timeline · resolve conflicts timeline · conflict-resolution · hypothesis-tracking
10
Gate: is the research exhaustive? research-exhaustiveness → GPS 5 threshold questions + 7-point stop criteria. If no, loop back to step 3.
11
Write the conclusion proof-conclusion → GPS tier + vehicle + narrative. Updates tree.gedcomx.json.

Researcher Profile

When you start a new project, init-project asks two short questions. The answers live in research.json and every skill reads them to adapt its behavior.

Question 1 — Experience level

  • Just starting out — step-by-step narration, terms defined, "why I'm doing this" for each action
  • Some research under my belt — moderate narration
  • Experienced — concise reporting, no hand-holding
  • Professional or certified — terse; full GPS vocabulary assumed

The level maps to a narration_guidance string in research.json. Every SKILL.md opens with an instruction to read and apply it.

Question 2 — Paid subscriptions

Ancestry, MyHeritage, FindMyPast, Newspapers.com, GenealogyBank, FindAGrave-Plus, or none.

Drives search-external-sites URL prioritization — subscribed sites land first, unsubscribed sites are still searchable but flagged.

Mid-session overrides

You can adjust narration on the fly without re-running the interview:

  • "Be more verbose" / "explain that step"
  • "Skip the explanations" / "just do it"
  • "Define genealogy terms when you use them"

How We Work Together

Pairing — Developer + Genealogist

  • Each genealogist is paired with one developer
  • Pairings rotate every week — over time you'll work with everyone on the team
  • The developer brings code skills; you bring domain expertise
  • Neither of you needs to know everything — that's the point
  • Ask questions freely. "Why does this work this way?" from a genealogist is exactly how we find gaps in the product

Claude licenses — today

Everyone gets a Claude license later today. This lets you install Cowork and run the plugin yourself. Claude Code (the CLI) is what developers use day-to-day; Cowork is the research environment end users interact with.

Fridays — Learning Days

No sprint work on Fridays. We step back and learn how to use AI effectively — prompting, evaluation, where AI helps and where it misleads. Share what you learned the following Monday.

This Week — Task 2: Deep-Dive a Skill

Each of the 10 developer-genealogist teams takes ownership of one skill this week. Your assignment is on the Kanban board — you'll be added to the board today.

Step 1 — Read & understand

  • Read SKILL.md in full — the instructions Claude follows
  • Read all docs in references/ — GPS standards, classification guides, place-handling rules, validation protocols, and more. These shape how Claude reasons inside the skill.
  • Read any templates in templates/ and scripts in scripts/ if present
  • Read existing tests, scenarios, and fixtures in eval/
  • Run the existing tests: make test

Step 2 — Use it on real research

  • Try the skill in Cowork with real genealogy inputs
  • Try edge cases — unusual names, missing dates, conflicting records
  • Try inputs that should fail gracefully
  • Note every place the output is wrong, confusing, or incomplete
  • Your genealogy eye is the quality bar — does this look right to you?

Step 3 — Improve & ship

  • Write new test scenarios and fixtures based on what you found
  • Improve SKILL.md and the supporting references/ docs — clearer instructions, better examples, missing edge cases
  • Fix issues you discovered
  • Make the skill, reference docs, and tests the best they can be
  • Open a PR by Friday

There is no constraint on what you improve. If you think the skill misses a common genealogy scenario, add it. If the instructions are confusing, rewrite them. You have ownership.

This Week's 10 Skills

One team per skill. Each skill covers a distinct part of the GPS workflow or reference layer — you'll learn the part of the system your skill touches most deeply.

translation

Genealogy-specific translation — German, French, Spanish, Italian, Dutch, Latin, Portuguese; period handwriting and abbreviations.

citation

Polish citations to Evidence Explained standards (Who/What/When/Where/Where-within).

convert-dates

Calendar boundary conversions — Julian/Gregorian, Old Style/New Style, Quaker double-dating.

historical-context

Explain boundary changes, naming conventions, migration patterns, and cultural context affecting records.

locality-guide

Structured research guide for a place/time — what records exist and where they're held.

check-warnings

Guardrail: flags genealogical impossibilities (died before born, married at 8, child after parent's death).

search-external-sites

Generates search URLs for Ancestry, MyHeritage, FindMyPast, FindAGrave, Newspapers.com.

search-wiki

Searches the FamilySearch Research Wiki for how-to guidance; saves findings as a markdown file.

search-wikipedia

Reference skill — fetches a Wikipedia summary and saves it as a markdown file. The canonical minimal example.

validate-schema

Validates both project files against published schemas — required fields, enum values, ID prefixes, cross-references.
